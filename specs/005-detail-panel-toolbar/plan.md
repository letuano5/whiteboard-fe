# Implementation Plan: Detail Panel & Basic Toolbar (P1A-07 + P1A-08)

**Branch**: `feat/local-editor` | **Date**: 2026-06-24 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `specs/005-detail-panel-toolbar/spec.md`

## Summary

P1A-07 adds a floating detail panel that appears when exactly one element is selected and allows real-time property editing via `patchElement`. P1A-08 adds a floating toolbar with 6 tool buttons (select, hand, rectangle, ellipse, line, text), the active tool visually highlighted, and all transient interaction state cleared on tool switch. Both components are already implemented and integrated into `Whiteboard.tsx`. This plan documents the verified implementation and the acceptance-test coverage work needed for spec `005-detail-panel-toolbar`.

## Technical Context

**Language/Version**: TypeScript 6 / React 19

**Primary Dependencies**: Zustand 5 (interaction.store, elements.store), Lucide React (toolbar icons), React Testing Library + Vitest 4

**Storage**: N/A (no new persistence; mutations go through `patchElement` which handles versioning)

**Testing**: Vitest 4 + React Testing Library; `pnpm test`

**Target Platform**: Web browser (Chromium + Firefox); rendered via SVG/DOM

**Project Type**: Single-page web app (frontend-only for P1A)

**Performance Goals**: Panel and toolbar updates within same render frame (≤16ms); no observable lag on selection change

**Constraints**: No Canvas overlay in P1A; all coords via camera transform; strict TypeScript

**Scale/Scope**: Single canvas, single user (P1A); sync comes in P2

## Constitution Check

| # | Principle | Status | Notes |
|---|-----------|--------|-------|
| I | Unified Element Store — renderer holds no state | ✅ | DetailPanel reads from `elements.store`; renders are pure views |
| II | Element Versioning — `version++`, new `versionNonce`, `updatedAt` on every mutate | ✅ | All prop changes go through `patchElement` (pipeline handles versioning) |
| III | Shared Camera Transform — all layers use `camera.store.ts` | ✅ | Panel and Toolbar are fixed-position UI overlays; no coordinate transform needed |
| IV | ShapeUtil Strategy — no type branching in core | ✅ | Panel does conditional rendering for `type==='text'` in UI only, not in core/pipeline |
| V | Sync Data Not Renderer | ✅ | Only `ElementProps` patches cross the pipeline; no render state leaves the client |
| VI | Single Mutation Pipeline | ✅ | DetailPanel uses only `patchElement`; Toolbar makes no element mutations |
| VII | Committed vs Transient State | ✅ | Panel reads `elements.store` (committed); Toolbar writes `interaction.store` (transient) |

No violations. No Complexity Tracking entries needed.
