# Implementation Plan: Remote Selection Highlight & Draft Preview

**Branch**: `feat/online-room` | **Date**: 2026-06-28 | **Spec**: [spec.md](./spec.md)

**Input**: Feature specification from `specs/018-remote-selection-highlight/spec.md`

## Summary

Implement P2.5-04: (1) broadcast each client's `selectedIds` alongside cursor-move so peers can render colored selection borders around elements another user has selected; (2) broadcast transient draft element state via a new `element-draft` event so peers can see ghost previews of ongoing drags/resizes/creates without writing to committed state.

All remote state (selection highlights, draft ghosts) lives exclusively in `interaction.store` per Constitution Principle VII. The server relays both events without storing them.

## Technical Context

**Language/Version**: TypeScript 5.x / 6.x (strict), Node 22 LTS
**Primary Dependencies**: React 19, Zustand 5, Socket.IO 4.8 (client + server), Vite 8
**Storage**: In-memory only (transient presence data, no persistence)
**Testing**: Vitest 4 (frontend unit tests)
**Target Platform**: Web (modern browsers, React SPA)
**Project Type**: Monorepo web app (frontend + backend + shared)
**Performance Goals**: Draft updates ≤ 50 ms throttle (20 FPS); selection highlights appear within 150 ms end-to-end
**Constraints**: No new third store; extend existing SvgLayer; TypeScript strict throughout
**Scale/Scope**: Up to ~10 concurrent peers per room at this phase

## Implementation Phases

### Phase A — Shared types & backend relay

1. **`packages/shared/src/index.ts`**: Add `ELEMENT_DRAFT: 'element-draft'` to `WS_EVENTS`.
2. **`backend/src/index.ts`**:
   - Extend `cursor-move` relay to forward `selectedIds` from the incoming payload to peers.
   - Add `element-draft` socket handler: relay `{ sessionId, elements }` to room peers (no storage).
   - Add `sessionId` to `ELEMENT_UPDATE` relay so receivers can identify who committed (enables draft cleanup).

### Phase B — Frontend state

3. **`frontend/src/types/interaction.ts`**: Add `remoteDrafts: Map<string, Element[]>` to `InteractionState`.
4. **`frontend/src/store/interaction.store.ts`**: Add `remoteDrafts: new Map()` to `DEFAULT_STATE`; add `setRemoteDrafts` action.

### Phase C — Frontend networking

5. **`frontend/src/sync/socket-client.ts`**:
   - **Emit `selectedIds` in cursor-move**: Subscribe to `interaction.store.selectedIds` changes; on change, emit `cursor-move` with current cursor (may be null), current viewport, and new `selectedIds`. Throttle at ≤ 50 ms.
   - **Receive `selectedIds`**: In the `CURSOR_MOVE` receiver, when `selectedIds` is present, update `remoteCursors[sessionId].selectedIds`.
   - **Subscribe to draft state**: Subscribe to `interaction.store` `draftElement` + `draftElements`; throttle at 50 ms; emit `ELEMENT_DRAFT` with the combined draft array. Emit `[]` when draft clears.
   - **Receive `element-draft`**: On `ELEMENT_DRAFT` event, call `setRemoteDrafts` — update or clear entry for `sessionId`.
   - **Clear drafts on commit**: In the `ELEMENT_UPDATE` receiver, if payload includes `sessionId`, clear `remoteDrafts` for that session.
   - **Clear drafts on leave**: In `USER_LEAVE` handler, also delete from `remoteDrafts`.

### Phase D — Rendering

6. **`frontend/src/canvas/layers/SvgLayer.tsx`**:
   - Read `remoteCursors` and `remoteDrafts` from `interaction.store`.
   - **Remote draft ghosts**: For each entry in `remoteDrafts`, find peer's color from `remoteCursors`, render each `Element` via `getShapeUtil(el.type).render(el)` at 0.5 opacity inside a `<g>`, preceded by a colored 1 px border `<rect>`. Render before local drafts in z-order.
   - **Remote selection highlights**: For each entry in `remoteCursors`, for each `selectedId`, find the element in `elements`, render a `<rect>` with solid stroke in the peer's color, no fill, no handles. Render above remote drafts but below local selection overlay.

### Phase E — Tests

7. **`frontend/src/sync/__tests__/socket-client.test.ts`**: Add tests covering:
   - `selectedIds` is emitted in cursor-move when `interaction.store.selectedIds` changes.
   - Incoming cursor-move with `selectedIds` updates `remoteCursors[sessionId].selectedIds`.
   - `element-draft` is emitted (throttled) when `draftElements` changes.
   - Incoming `element-draft` updates `remoteDrafts`.
   - Empty `element-draft` (`elements: []`) clears `remoteDrafts[sessionId]`.
   - Draft cleared on `USER_LEAVE`.
8. **`frontend/src/canvas/layers/__tests__/SvgLayer.test.tsx`**: Add tests covering:
   - Remote selection border renders for a peer's selected element in that peer's color.
   - Remote selection border does NOT render handles.
   - Remote draft ghost renders at reduced opacity.
   - Nothing renders when `remoteCursors` / `remoteDrafts` are empty.

## Open Questions / Risks

| Risk | Mitigation |
|------|-----------|
| `ELEMENT_UPDATE` relay doesn't include `sessionId` today — can't reliably clear remote draft on commit | Add `sessionId` to relay payload in backend Phase A |
| Selection change without cursor movement (keyboard Ctrl+A) won't trigger existing `emitCursorMove` | Separate subscription to `selectedIds` in socket-client (Phase C) |
| Multiple peers selecting same element — colored borders overlap | Accept overlap; different colors make it identifiable; no offset needed at this scale |
