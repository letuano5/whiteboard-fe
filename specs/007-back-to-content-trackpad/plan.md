# Implementation Plan: Back to Content & Trackpad Support

**Branch**: `feat/local-editor` | **Date**: 2026-06-24 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `specs/007-back-to-content-trackpad/spec.md`

## Summary

Add three canvas navigation improvements to the whiteboard:
1. A "Back to content" floating button that appears when all shapes are off-screen and navigates the camera to fit all content.
2. Trackpad two-finger scroll pans the canvas (no zoom); Ctrl/Cmd + scroll or pinch gesture zooms with reduced sensitivity.
3. A small hint text visible in Select mode only.

All changes are pure event-handling and UI overlay — no new Element types, no mutation pipeline changes, no new stores.

## Phase 0: Research

### Research decisions

**Decision 1 — Wheel event zoom/pan distinction**

Browsers set `WheelEvent.ctrlKey = true` for both:
- Ctrl/Cmd + mouse wheel
- Trackpad pinch gesture (synthetic ctrlKey injected by browser)

This is the reliable, browser-standard way to distinguish "user wants to zoom" from "user wants to pan". No other mechanism (delta size, `isPinch`) is needed.

**Decision 2 — Zoom sensitivity formula**

Use `Math.exp(-normalizedDelta * ZOOM_SENSITIVITY)` with `ZOOM_SENSITIVITY = 0.001`:

| Input source | Typical deltaY | Factor per event | Effective change |
|---|---|---|---|
| Mouse wheel notch | ~100 | exp(-0.1) ≈ 0.905 | ~9.5% per click |
| Trackpad pinch tick | ~3 | exp(-0.003) ≈ 0.997 | ~0.3% per tick |

This satisfies AC-8 (≤ 0.01 per raw unit) and preserves familiar mouse-wheel feel while making trackpad pinch smooth.

**Decision 3 — deltaMode normalization**

`WheelEvent.deltaMode` values: 0=PIXEL (default), 1=LINE (≈16px), 2=PAGE (≈viewport size).
Multiply accordingly before applying sensitivity or pan offset.

**Decision 4 — Viewport size for back-to-content**

Read `containerRef.current.getBoundingClientRect()` at the time of:
- Rendering the button (inside a `useMemo` recomputed on camera/elements change)
- Computing the fit camera when the button is clicked

This avoids a ResizeObserver for now; acceptable since the container fills the window.

**Decision 5 — Fit-to-content camera formula**

Given content bounding box `(minX, minY, maxX, maxY)` and viewport `(W, H)`:
```
zoom = clamp(min(W * PAD / contentW, H * PAD / contentH), MIN_ZOOM, MAX_ZOOM)
camX = (minX + maxX) / 2 - W / (2 * zoom)
camY = (minY + maxY) / 2 - H / (2 * zoom)
```
Where `PAD = 0.85` (leaves 15% total margin, ~7.5% each side).

**Decision 6 — SelectHint placement**

Fixed bottom-left overlay inside the canvas container, low z-index, subtle muted color.
No interaction needed; display depends solely on `tool === 'select'`.

**Decision 7 — BackToContent placement**

Fixed bottom-center overlay (absolute positioning inside the container div), placed directly above the bottom toolbar with a small gap so the two controls never overlap.
Visible only when: `elements.some(e => !e.isDeleted)` AND no non-deleted element intersects the viewport.
