# Implementation Plan: Reconnect Without Data Loss (P3A-03)

**Branch**: `feat/online-room` | **Date**: 2026-06-29 | **Spec**: [spec.md](./spec.md)

**Input**: Feature specification from `specs/024-reconnect-diff/spec.md`

## Summary

Add a reconnect-diff protocol to Socket.IO: on reconnect the client sends its last known
`documentClock`; the server computes and returns only the changed/deleted elements since that
clock (`ROOM_DIFF`), falling back to a full snapshot when tombstone history is insufficient.
The client applies the diff and then replays any locally queued mutations from the disconnected
period.

joins (lastServerClock = 0). Must not block the socket event loop for reconnect diff computation.

**Scale/Scope**: Single-room diff; no horizontal scaling concern (Redis deferred to P3D).

## Phase 0: Research

### R-01 — Socket.IO reconnect event model

**Decision**: Use `socket.on('connect', handler)` which fires on both initial connection and
every reconnect. Track `_hasJoined: boolean` to distinguish first join from reconnect.

**Rationale**: Socket.IO 4.x does not expose a dedicated "reconnect" event on the socket object
itself (only on `socket.io`). Using the `connect` event + a flag is the idiomatic approach.
On reconnect, `connect` fires with `socket.recovered = false` (since we don't use connection
state recovery). Sending JOIN_ROOM with the current `_lastServerClock` on every `connect`
unifies the code path.

**Alternatives considered**:
- `socket.io.on('reconnect')`: fires on `socket.io` (the Manager), not on the socket instance
  itself; harder to access `_lastServerClock` cleanly from within the module.
- `socket.recovered`: Socket.IO's built-in connection state recovery — requires server-side
  session storage we don't have; out of scope.

### R-02 — Pending queue vs Socket.IO internal buffering

**Decision**: Check `_socket.connected` before calling `_socket.emit` in the mutation hook.
When disconnected, push elements to `_pendingQueue: Element[]` instead of emitting. After
applying ROOM_DIFF, flush the queue with a single ELEMENT_UPDATE.

**Rationale**: Socket.IO buffers emissions made while disconnected by default (auto-reconnect
mode). If we push to `_pendingQueue` AND also call `_socket.emit`, the event would be
double-sent. By guarding `_socket.emit` with the `connected` check, we suppress Socket.IO's
internal buffer for our mutations and control ordering ourselves.

**Alternatives considered**:
- `socket.volatile.emit`: drops the event if disconnected (no buffering), no queue; insufficient
  because we need offline mutations to reach the server eventually.
- Disabling Socket.IO's internal buffer via `retries: 0`: global option; side-effects on
  other event types (cursor, presence) that we don't need to queue.

### R-03 — Diff computation: DB vs in-memory

**Decision**: Query the DB for changed records (`recordClock > lastServerClock`) and tombstones
(`deletedClock > lastServerClock`). Then overlay any in-memory element that is NOT already in
the DB changed set (to capture not-yet-autosaved mutations).

**Rationale**: Per-element clock (`recordClock`, `deletedClock`) is only available in the DB.
In-memory elements have no individual timestamps. The overlay step conservatively adds any
element that is in memory but not covered by the DB diff; LWW on the client handles duplicates.
The autosave runs every 5 s; brief reconnects (< 5 s) may produce a non-empty overlay; longer
reconnects will have a near-empty overlay.

**Alternatives considered**:
- Flush (autosave) before computing diff: adds latency to every reconnect; complicates
  autosave lifecycle (flush-while-room-active); rejected.
- Full in-memory snapshot as diff: defeats SC-002 (bandwidth proportional to changes);
  rejected.

### R-04 — tombstoneHistoryStartsAtClock: column vs dynamic

**Decision**: Compute dynamically via `MIN(tombstone.deletedClock)` at reconnect time (per
FR-003). The `Room.tombstoneHistoryStartsAtClock` DB column stays unused in P3A (tombstones
are never pruned). Tests that need the wipe-all path will insert a tombstone with a high
`deletedClock` and use a low `lastServerClock`.

**Rationale**: FR-003 explicitly says "smallest deletedClock among all tombstones". Using the
DB column (always 0 in P3A) would make the wipe-all path unreachable. The dynamic MIN query
is a single indexed lookup on `(roomId, deletedClock)` — cheap.
