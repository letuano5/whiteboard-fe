## Implementation Slices

1. Inspect existing shared sync contracts and backend `SyncRoom` planning extension points.
   - Verification: no source edits until target APIs are identified.

2. Add backend validation and conflict planning for `PatchSlotsCommand`.
   - Enforce slot clocks, invalid fields, invalid slots for element type, atomic rejection, and
     same-slot latest-to-server behavior.
   - Verification: `pnpm --filter whiteboard-be test -- sync-room`.

3. Add delete-wins, viewer-boundary rejection, reference validation, and command limits.
   - Reject later patches to deleted records, reject viewer actors before planning, inject
     asset/group/frame reference checks, and enforce patch/delete/repair/change-set limits.
   - Verification: AC-tagged backend tests.

4. Add linear geometry validation and server-side bbox normalization.
   - Reject unsupported `transform.*` patches for linear element types and normalize bbox from
     valid geometry slots.
   - Verification: AC-tagged backend tests.

5. Export updated sync primitives and document verification.
   - Verification: AC coverage script, backend sync tests, backend typecheck, repo typecheck.

## Risks And Deviation Handling

- Risk: existing P5-02 shared validators already reject a case needed for server-side conflict
  planning. Mitigation: extend shared validation narrowly and keep backend conflict semantics in
  `SyncRoom`.
- Risk: asset/group/frame validation needs storage data not available in hot state. Mitigation:
  use injectable validators in backend tests and leave binary asset lifecycle to later phases.
- Risk: ACK/rebase payload shape belongs to P5-05. Mitigation: return minimal backend result
  metadata only where needed; do not wire socket broadcast protocol in this phase.
- If implementation requires persistence transactions or durable idempotency, stop and defer that
  concern to P5-06 instead of expanding P5-04.
