# Plan 05.6-02: Add SyncRoom transactional persistence and idempotency

## Scope

Integrate optional persistence into `SyncRoom`: persisted duplicate lookup before validation,
conditional one-clock transaction commit, record/tombstone/slot-clock invariants, memory mutation
only after commit, post-commit unhealthy reload, and rebuilt hot maps/indexes before resuming.

## Target Files

- `backend/src/sync/sync-room-contracts.ts`
- `backend/src/sync/sync-room-errors.ts`
- `backend/src/sync/sync-room-payload-hash.ts`
- `backend/src/sync/sync-room-persistence.ts`
- `backend/src/sync/sync-room.ts`
- `backend/src/sync/index.ts`
- `backend/src/sync/sync-room-persistence.test.ts`

## Implementation Slices

1. Add persistence interfaces for processed request lookup, conditional commit, reload state, and
   command persistence policy.
   - Verification: focused typecheck of backend sync tests.

2. Implement canonical command payload hashing that excludes actor/debug/transient metadata, and
   ensure duplicate lookup runs before domain validation.
   - Verification: tests for replayed create retry and duplicate payload conflict.

3. Persist accepted change sets with a single conditional `Room.documentClock` update and with
   touched `recordClock`, `deletedClock`, and touched `Record.slotClocks` sharing that same
   `documentClock`; for every written record, enforce `recordClock = max(slotClocks[*].clock)`.
   - Verification: persistence adapter tests inspect transaction writes.

4. Make DB commit happen before hot memory apply; DB failure leaves memory unchanged and no
   `afterApply` broadcast hook runs.
   - Verification: `AC-6` unit test.

5. On conditional clock conflict or post-commit `applyCommitted` failure, mark the room unhealthy,
   reload from Postgres state, rebuild hot maps/indexes (`elements`, `slotClocks`,
   `tombstoneElementIds`, processed request cache), then resume accepting commands.
   - Verification: `AC-7`/`AC-8` unit tests.

## Acceptance-Test Tasks

- `AC-1`: same request/payload replay returns old result without clock increment.
- `AC-2`: same request/different payload rejects `DUPLICATE_REQUEST_CONFLICT`.
- `AC-3`: duplicate replay skips `afterApply` broadcast work.
- `AC-4`: new hot room can replay persisted processed request after memory loss.
- `AC-5`: multi-delete/multi-repair change set uses one `documentClock`.
- `AC-6`: DB failure leaves memory unchanged and skips broadcast hook.
- `AC-7`: post-commit apply failure reloads and rebuilds hot maps before resume.
- `AC-8`: conditional clock conflict reloads and returns unhealthy failure without ACK-worthy result.
- `AC-9`: transient patch policy is relaxed; final/discrete policy is durable.
- `AC-10`: resendable commands cannot skip `ProcessedRequest`.

## Risks And Deviation Handling

- Risk: post-commit apply failure is rare in production. Mitigation: exercise it through a controlled
  test hook while keeping production recovery path explicit.
- Risk: persistence adapter could drop old slot clocks. Mitigation: commit tests must verify full
  per-record slot clock JSON and `recordClock = max(slotClocks[*].clock)`.
