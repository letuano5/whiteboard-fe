# Phase 2.57 тАФ UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Scope note:** This phase adds zero new screens/pages. All UI surface is: (1) two new items in
the existing right-click `ContextMenu`, (2) two new keyboard shortcuts (no visible chrome), (3) a
new "group selection" SVG overlay variant on the existing canvas, and (4) new bound-text rendering
inside the existing SVG `<text>` shape. Every visual decision below is a **reuse-and-extend** of
already-shipped canvas-overlay/context-menu conventions in this codebase, not a new visual
language тАФ per the explicit instruction in the phase brief to match existing style rather than
invent one.

| Property | Value |

| Tool | none |

context menus). Initializing shadcn now, for a phase that only adds two menu rows and one SVG
overlay variant, would contradict the phase brief's explicit "match existing visual style ... 
rather than inventing a new visual language" instruction and is out of scope for this phase.

## Typography

This phase introduces no new typographic role. It reuses two already-declared type styles
verbatim and does not add a new font-size scale:

| Role | Size | Weight | Line Height | Source |
|------|------|--------|-------------|--------|
| Context menu item | 13px | 400 (regular; no `font-weight` override in `BUTTON_STYLE`) | normal (browser default, ~1.2) | `ContextMenu.tsx` `BUTTON_STYLE.fontSize` тАФ reused as-is for the new Merge/Unmerge rows |
| Canvas text element (incl. bound/wrapped text) | user-controlled `props.fontSize` (default `16`) | 400 (no bold toggle exists yet) | `fontSize * 1.2` | `canvas/shapes/text.tsx` L11-12 тАФ bound text uses the SAME per-element `fontSize`/`fontFamily`/line-height formula as any other text element; this phase must NOT introduce a separate "bound label" font scale |

the element's own `props.fontSize` or `props.fontFamily`. (D-03 in CONTEXT.md separately forbids
scaling `fontSize` for *independent*, non-bound group text on group-bbox resize; bound text's
`fontSize` is likewise never auto-scaled by this phase тАФ only re-wrapped/re-centered.)

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA (Merge) | `Merge` тАФ context menu row label. Enabled when 2+ selectable (non-locked, per D-02) elements are selected. Disabled (existing `DISABLED_STYLE`) otherwise, including when the selection contains only locked elements. |
| Primary CTA (Unmerge) | `Unmerge` тАФ context menu row label. Enabled when the current selection resolves to an existing group (`groupId !== null` on the clicked/selected element). Disabled (existing `DISABLED_STYLE`) otherwise. |
| Keyboard shortcut hint | none shown in the menu row тАФ matches the existing convention where `Bring to Front` / `Forward` / `Backward` / `Send to Back` rows show no shortcut-hint suffix today. | Empty state | not applicable тАФ this phase adds no list/empty-data view |
| Error state | not applicable тАФ merge/unmerge/group-cascade operations cannot fail from user input in a way that needs an error message (they are pure local mutations over the current selection); if the planner discovers a genuine failure path, it must reuse whatever error-surface convention already exists elsewhere in `canvas/tools/select/` rather than inventing one here |
| Destructive confirmation | none. Group delete-cascade (AC-10) and Unmerge (AC-2) execute immediately with no confirmation dialog, exactly matching this app's existing pattern for every other canvas delete path (Delete key, eraser tool) тАФ all of which rely on the existing local undo history (`Cmd/Ctrl+Z`) instead of a confirm modal. Do not introduce a confirmation dialog for this phase; it would be the first of its kind in the canvas tool surface and is out of scope. |

## Component Contracts (phase-specific)

### 1. Context Menu тАФ Merge / Unmerge rows

**File to modify:** `frontend/src/components/context-menu/ContextMenu.tsx`

- Reuse `BUTTON_STYLE` / `DISABLED_STYLE` and the existing `onMouseEnter`/`onMouseLeave` inline
  hover-background pattern (`#f3f4f6` on hover, `none` otherwise) verbatim for the two new rows тАФ
  do not create a second button style.
- Remove/relax the current `const disabled = selectedCount !== 1 || !selectedId` gate for z-order
  items only as needed to keep z-order commands single-selection-only (unchanged z-order
  behavior); Merge/Unmerge get their OWN independent enabled/disabled computation, not the
  single-selection gate (per RESEARCH.md's explicit instruction to lift the gate for Merge).
- **Layout:** append a divider (`borderTop: '1px solid #e5e7eb'`, `margin: '4px 0'`, no visible
  label) below the existing 4 z-order rows, then the `Merge` row, then the `Unmerge` row, in that
  order. This is the one new visual primitive this phase introduces (a divider), reusing the
  existing border color token (`#e5e7eb`) already used for the menu's own outer border тАФ no new
  color is introduced.
- Container chrome (`background`, `border`, `borderRadius: '6px'`, `boxShadow`, `padding: '4px 0'`,
  `minWidth: '160px'`, `zIndex: 9999`) is unchanged.

### 2. Group Selection Overlay (new тАФ D-01 trigger + AC-9 resize)

**Files likely touched:** `frontend/src/canvas/layers/svg/SelectionOverlay.tsx`,
`LocalSelectionLayer.tsx`, `selectors.ts` (per RESEARCH.md's `group.ts`/`group-resize.ts` plan).

- **Trigger:** clicking any element whose `groupId` is non-null selects the whole group (D-01).
  The rendered outline for this state MUST look identical to the existing ad-hoc multi-select
  outline: `stroke="#3b82f6"`, `strokeWidth={1.5}`, `strokeDasharray="4 2"`, `fill="none"`,
  computed via `getMultiSelectBounds`-style min/max bbox math (reuse `getMultiSelectBounds`,
  passing resolved group member ids instead of `selectedIds`). Do not introduce a new stroke color,
  width, or dash pattern to visually distinguish "group" from "ad-hoc multi-select" тАФ CONTEXT.md
  explicitly says these must match.
- **New capability this phase adds (AC-9): resize handles on the group bbox.** The existing
  `MultiSelectionOverlay` component renders outline-only, with NO handles, because ad-hoc
  multi-select currently has no resize-by-bbox feature. Group-bbox resize is new in this phase, so
  the group-selection overlay needs 8 handles (`nw, ne, sw, se, n, s, e, w` тАФ same handle set as
  the single-element `SelectionOverlay` for non-image shapes) rendered at the bbox corners/edges,
  using the EXACT same handle visual spec as `SelectionOverlay`'s existing resize handles: `r={4}`,
  `fill="white"`, `stroke="#3b82f6"`, `strokeWidth={1.5}`, `cursor: 'pointer'`.
- **No rotate handle.** Per CONTEXT.md/RESEARCH.md, "rotate-by-bbox" is explicitly out of scope тАФ
  do not add the rotate-handle line/circle (`ROTATE_HANDLE_OFFSET` pattern) to the group overlay.
- **Scope boundary:** this new handled-overlay variant applies ONLY when the selection resolves to
  a shared `groupId` (a merged group). Plain ad-hoc multi-select (2+ elements selected without a
  common `groupId`) keeps today's outline-only `MultiSelectionOverlay` behavior unchanged тАФ do not
  retrofit resize handles onto ad-hoc multi-select as a side effect of this work.

### 3. Bound Text Layout (new тАФ AC-4/AC-6/AC-9 centered, wrapped label)

**File likely touched:** `frontend/src/canvas/text/text-wrap.ts` (new), `canvas/shapes/text.tsx`
(rendering extension).

- **Padding:** `8px` inset on each side of the container's bbox (the `sm` spacing token). Wrap
  width = `container.width - 16` (8px ├Ч 2). If this yields a wrap width below a sane minimum
  (`24px` floor, to avoid pathological one-character-per-line wrapping in very small containers),
  clamp wrap width to `24px` rather than allowing a zero/negative width.
- **Horizontal alignment:** centered тАФ text anchor / per-line x position centers each wrapped line
  within the padded width, matching how the existing single-line text render already centers via
  its `textX` computation in `text.tsx` (reuse the same anchor approach, do not invent a new one).
- **Vertical alignment:** centered as one block тАФ compute total block height as
  `lines.length * lineHeight` (reusing the existing `lineHeight = fontSize * 1.2` convention from
  `text.tsx` L12), then vertically center that block within the container's height. Do not
  introduce alternate vertical-alignment modes (top/bottom) тАФ CONTEXT.md/deferred list explicitly
  scopes this to single-region center-only.
- **Z-index:** bound text always renders with a `zIndex` greater than its container's current
  `zIndex` so the label paints above the shape. The exact renumbering mechanics (fixed `+1` vs.
  broader z-order recompute on cascade) are a planner/executor implementation choice тАФ the only
  user-observable visual contract is "label is always on top of its own container," including
  after any z-order command is applied to the container (AC-7).
- **Typography:** uses the bound text element's own existing `props.fontSize` / `props.fontFamily`
  unchanged (see Typography section above) тАФ wrapping/centering is a geometry change, never a font
  styling change.
