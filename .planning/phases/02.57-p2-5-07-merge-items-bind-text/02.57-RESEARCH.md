## User Constraints (from CONTEXT.md)

### Locked Decisions

**Group selection UX**
- **D-01:** Clicking one element that belongs to a group selects the **whole group** as one unit
  — combined bbox + move/resize handles appear, matching Figma-style grouped selection. This is
  the trigger mechanism for the AC-9 group-bbox resize operation. A separate interaction (e.g.
  double-click) is needed to drill into and select/edit a single element inside the group; the
  planner should define the exact drill-in gesture during planning/implementation (not re-asked
  here since it's a standard, low-ambiguity affordance).

**Locked elements + merge**
- **D-02:** A `locked` element cannot be added to a Merge selection — Merge is a no-op / excluded
  for locked elements. If an element inside an existing group becomes locked afterward, group
  cascade (move/delete) must skip that locked member — it stays in place and is not deleted when
  the rest of the group moves/deletes.

**Independent text font scaling on group resize**
- **D-03:** Text elements that are part of a group but **not** bound to a container (i.e. no
  container-eligible sibling to bind to) do **not** scale `fontSize` when the group is resized by
  its combined bbox. Only their `(x, y)` position is translated/scaled to the new bbox; `fontSize`
  stays fixed. This applies only to independent group text — container-bound text still
  recenters/rewraps per AC-6/AC-9 (that binding behavior is unaffected by this decision).

**Plan slicing (delivery cadence)**
- **D-04:** `gsd-plan-phase` must structure this phase as **3 sequential plan slices**, matching
  the sub-phase breakdown already recommended in `docs/SPECS.md` `[P2.5-07]` "Ghi chú triển khai":
  1. **Slice (a):** Merge/Unmerge commands (context menu + Cmd/Ctrl+G, Cmd/Ctrl+Shift+G), join-
     existing-group vs flatten-multi-group rule, group-aware delete cascade, and the copy/
     duplicate `groupId` remap fix (duplicate currently carries `groupId` through unchanged —
     `frontend/src/canvas/tools/select/clipboard.ts` `cloneAsNewDraft` L17 — must instead assign a
     fresh shared `groupId` to duplicated group members).
  2. **Slice (b):** Text-binding at merge time — center/wrap text into exactly one container,
     `zIndex` above the container, z-order command propagation to bound text, move/resize/delete
     following behavior for the bound pair.
  3. **Slice (c):** Group-bbox resize for multi-element groups — non-text scaling, container-bound
     text recenter/rewrap, independent text position-only scaling per D-03.
  Each slice should be its own plan/commit so it can be reviewed and rolled back independently.

## Summary

This phase adds a `groupId`-based "merge" unit-of-operation on top of the existing single/multi
select interaction code, plus one special case: binding exactly one `text` element to exactly one
container-eligible shape as a centered, wrapped label. Nothing here requires a new sync event,
new shared type, or new npm package — `Element.groupId` and its sync slot
(`grouping.groupId`) already exist end-to-end, and the mutation pipeline already supports
committing an array of patches as a single batch (which is also how the existing history
capture hook turns a batch into one undo/redo step — confirmed by reading
`frontend/src/sync/history-capture.ts` and `frontend/src/store/history.store.ts`).

The most consequential design decision the planner must lock down explicitly is: **binding is a
derived relationship, not stored state.** There is no `boundContainerId` field on `Element` and
none should be added (AC-13 forbids new sync slots) — a text element counts as "bound" only when,
at evaluation time, its `groupId` group has exactly one `text` member and exactly one
container-eligible member (AC-4 vs AC-5). Every operation that must special-case bound text
(move, resize, delete-cascade, z-order, group-bbox-resize, duplicate) needs to re-derive this
relationship from the current element list, ideally through one shared selector
(e.g. `resolveGroupBinding(groupId, elements)`) reused by all of them — this is the single most
important "don't hand-roll it five times" risk in this phase.

The second load-bearing finding: this codebase has **no canvas 2D context available in tests**
(jsdom, no `canvas` npm package, no `getContext` mock in `src/test/setup.ts`). Any text-wrap
implementation that calls `canvasEl.getContext('2d').measureText()` unconditionally will return
`null` from `getContext` under Vitest/jsdom and crash. The wrap utility must be written test-safely
(inject/override the measurement function, or guard for a null context with a monospace-width
fallback) — this is a concrete, verified-in-this-repo pitfall, not a hypothetical one.

**Primary recommendation:** Implement text wrapping with canvas 2D `measureText()` word-width
measurement + manual greedy line-splitting + one `<tspan>` per line (extending the existing
`textShapeUtil.render` tspan-per-line pattern already in the codebase), NOT `foreignObject`. Keep
wrap-width computation a single flat inset (`container.width - 2*padding`) regardless of container
shape type — SPECS.md explicitly scopes out per-shape auto-layout complexity, so there's no need
to replicate Excalidraw's per-shape-type (rect/ellipse/diamond) inset math.

### Pattern 1: Derived binding resolution (single source of truth)
**What:** One function, called everywhere a container/text pair's binding status matters, that
inspects the current `elements` array and returns whether a given `groupId` is a "bound pair" per
AC-4/AC-5 rules (exactly 1 text + exactly 1 container-eligible shape in that group).
**When to use:** move/resize/delete cascade, z-order propagation, group-bbox resize, duplicate.
**Example (illustrative, not copied from a source — this is new code for this repo):**
```typescript
// frontend/src/canvas/tools/select/group.ts
const CONTAINER_TYPES = new Set<ElementType>([
  'rectangle', 'ellipse', 'diamond', 'triangle', 'polygon', 'image',
]);

export interface GroupBinding {
  textId: string;
  containerId: string;
}

export function resolveGroupMembers(groupId: string, elements: Element[]): Element[] {
  return elements.filter((el) => !el.isDeleted && el.groupId === groupId);
}

// @covers AC-4, AC-5: binding only when exactly 1 text + exactly 1 container-eligible shape
export function resolveGroupBinding(groupId: string, elements: Element[]): GroupBinding | null {
  const members = resolveGroupMembers(groupId, elements);
  const texts = members.filter((el) => el.type === 'text');
  const containers = members.filter((el) => CONTAINER_TYPES.has(el.type));
  if (texts.length !== 1 || containers.length !== 1) return null;
  return { textId: texts[0].id, containerId: containers[0].id };
}
```

### Pattern 2: Merge rule (join existing group vs flatten)
**What:** AC-3's rule: if the selection touches exactly one existing `groupId` (mixed with
ungrouped elements), reuse it; if it touches 2+ distinct existing `groupId`s, generate one new id
for everyone.
**Example:**
```typescript
// frontend/src/canvas/tools/select/merge.ts
export function resolveMergedGroupId(selected: Element[]): string {
  const existingGroupIds = new Set(
    selected.map((el) => el.groupId).filter((id): id is string => id !== null),
  );
  if (existingGroupIds.size === 1) return [...existingGroupIds][0];
  return generateId(); // 0 or 2+ distinct existing groups -> fresh id
}
```

### Anti-Patterns to Avoid
- **Storing "isBoundText"/"boundContainerId" as a new field:** AC-13 requires zero new sync slots.
  Compute it every time from `groupId` + type composition (Pattern 1). A stored flag can drift out
  of sync with group membership changes (e.g., after unmerge) and is an extra thing to keep
  consistent across duplicate/undo/redo.
- **Per-element `updateElements` calls in a loop for group operations:** breaks AC-12 (must be
  ONE undo step). Always collect all group member patches into a single array and call
  `updateElements(...)` / `deleteElements(...)` once. This mirrors the existing `computeMultiDragDrafts`
  pattern, which already does this for bound-arrow cascades.
- **Calling `canvas.getContext('2d')` without a null-guard in the wrap utility:** crashes under
  jsdom/Vitest (see Common Pitfalls below). Always accept an injectable measure function or check
  for `null` context.
- **Locking font measurement to a single hardcoded font-family assumption:** `Element.props.fontFamily`
  is already a free-form string (`text-editor.tsx` uses `element.props.fontFamily ?? 'sans-serif'`);
  the wrap function must use the actual element's fontFamily/fontSize when building the canvas font
  string (`getFontString`-equivalent), not a fixed default, or wrap width will be wrong whenever a
  non-default font is used.

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Combined bbox of multiple elements | A second bbox-computation function | `getMultiSelectBounds` (`canvas/layers/svg/selectors.ts` L89-109) — pass group member ids instead of `selectedIds` | Already handles min/max over x/y/width/height; reuse directly for AC-9 |

| Text width measurement | A DOM-based "render into hidden span, read scrollWidth" hack (as `text-editor.tsx` currently uses for the *editing* box) | Canvas 2D `measureText()` (Excalidraw's approach) for the *wrap calculation*, decoupled from the WYSIWYG editing box | DOM reflow measurement is slower and couples wrap logic to the DOM; canvas measurement is the standard approach for offline/render-time layout calculations and is synchronous/pure |

**Key insight:** almost every "propagate an operation across group members" problem in this phase
(move, resize, delete, z-order, duplicate) is structurally the same shape as the arrow-binding
propagation problem this codebase already solved in `bound-arrows.ts`. Treat that file as the
reference implementation pattern, not just prior art to read once.

## Common Pitfalls

### Pitfall 1: `canvas.getContext('2d')` returns `null` under Vitest/jsdom
**What goes wrong:** Calling `.measureText()` on a null context throws a "Cannot read properties
of null" error, but only in tests — the same code works fine in a real browser.
**Why it happens:** This project's test environment is `jsdom` (confirmed in
`frontend/vite.config.ts` `test.environment: 'jsdom'`), and jsdom does not implement the Canvas 2D
rendering context by default (`HTMLCanvasElement.getContext('2d')` returns `null` unless the
`canvas` npm package is installed and jsdom is configured to use it). This project has neither the
`canvas` package nor any `getContext` mock in `frontend/src/test/setup.ts` (verified — only a
`ResizeObserver` polyfill exists there).
**How to avoid:** Design the wrap/measure function to accept an injectable `measureTextWidth`
function (default: real canvas measurement; test-time: a deterministic character-width heuristic
or a small manual mock), OR add a lightweight `getContext('2d')` mock in `src/test/setup.ts`
alongside the existing `ResizeObserver` polyfill (approximate character-width based measurement is
sufficient for unit tests that assert on line-count/line-content, not pixel-perfect width).
**Warning signs:** Any new test file for the wrap utility failing with a null-context TypeError
the moment it's added to the Vitest suite.

### Pitfall 2: Treating group binding as stored state instead of derived state
**What goes wrong:** If a `boundContainerId`/`isBoundText` field is added to `ElementProps` (even
informally, not synced), it can go stale — e.g. after Unmerge (AC-2) the group is gone but the
flag lingers, or after merging in a third container the AC-5 "no binding" rule should apply but
the stale flag says otherwise.
**Why it happens:** It is tempting to cache the binding relationship at merge time for
performance, but the derivation (`resolveGroupBinding`, Pattern 1) is cheap (filter over a handful
of group members) and the source of truth is fully expressed by `groupId` + `type` already.
**How to avoid:** Always recompute via the shared selector at the point of use; never persist the
derived fact as element state.
**Warning signs:** Binding behavior (recenter/rewrap/z-order-follow) still applying after Unmerge,
or not applying immediately after a merge that changes group composition without a page reload.

### Pitfall 3: Locked-member exclusion silently dropped (D-02)
**What goes wrong:** Group move/delete cascades iterate "all elements with this `groupId`" without
checking `element.locked`, so a locked member gets dragged/deleted along with the rest of the
group, violating D-02.
**Why it happens:** The existing `computeMultiDragDrafts`/`computeBoundArrowDrafts` patterns filter
on `!el.isDeleted` but have no precedent for a `locked` filter (locked-element handling doesn't
exist yet in those flows since single/multi selection generally shouldn't select locked elements
in the first place — verify current locked-element selection behavior during planning, as it may
already be excluded upstream at pointer-down).
**How to avoid:** Any new "resolve full group membership" helper must filter out `locked` members
from the *move/delete* cascade set (but D-02 doesn't exempt locked members from *merge selection
eligibility checks* — a locked element simply can't be added to a Merge selection to begin with).
**Warning signs:** Locked shape moves or disappears when its group is dragged/deleted.

### Pitfall 4: `resize.ts` container-type check needs widening, not text-only
**What goes wrong:** `computeResizeProps` in `resize.ts` currently special-cases `el.type === 'text'`
(font-scale-to-fit) and `el.type === 'image'` (aspect-ratio lock). Group-bbox resize (AC-9, slice c)
is a *different* code path (multiple elements resizing together via combined bbox, not the existing
single-element resize session), so this existing function does not need modification for slice (c)
— but the planner should verify during slice (c) design that group-resize is built as its own
function (per the Architecture diagram: `group-resize.ts`) rather than accidentally overloading
`resizeSession`/`computeResizeProps`, which are wired to the single-drag-handle interaction model.
**Why it happens:** The existing single-element resize session (`resizeSession`,
`resizeBoundsFromAnchorAndPointer`) is anchor/handle-driven; group-bbox resize is a from-old-bbox
to new-bbox linear transform applied to N elements, a structurally different computation.
**How to avoid:** Treat group-bbox resize as new math (proportional scale of each member's x/y/width/height
relative to the old combined bbox), not an extension of per-element resize sessions.
**Warning signs:** Attempting to reuse `ResizeSession`/`resizeHandle` state for multi-element group
resize and hitting anchor-math bugs meant for single-element rotation-aware resize.

Group move/resize/merge/unmerge/z-order-cascade must all collect every affected element's patch
into one array and call this once — never once per element — to satisfy AC-12.

## Open Questions

1. **Exact drill-in gesture to select a single element inside an already-selected group**
   - What we know: CONTEXT.md explicitly defers this to planner discretion (D-01 discretion note);
     double-click is the most standard cross-app convention (Figma, Excalidraw both use
     double-click-to-drill-into-group).
   - What's unclear: whether double-click conflicts with the existing `onCanvasDoubleClick` handler
     in `text-editor.tsx`, which currently double-click-hit-tests only `text`-type elements to enter
     edit mode. A grouped `text` element being double-clicked needs disambiguation: drill-into-group
     vs enter-text-edit-mode.
   - Recommendation: Plan should specify that double-click on a grouped **text** element still
     enters edit mode (existing higher-priority behavior, AC not being changed), while double-click
     on a grouped **non-text** element drills into single-element selection within the group. This
     preserves existing text-editing UX without a new modifier key.

2. **Whether font-size independent-text scaling in AC-9/D-03 needs a minimum/maximum clamp**
   - What we know: D-03 says independent text position translates/scales but `fontSize` stays
     fixed; SPECS.md's own wording ("có thể scale tuyến tính theo group resize nếu không làm chữ
     tràn xấu") suggested optional linear scaling, but CONTEXT.md's D-03 explicitly overrides this
     to "no fontSize scaling, position only."
   - What's unclear: nothing structurally — D-03 is unambiguous and supersedes the SPECS.md
     wording. Flagging only so the planner doesn't accidentally implement the older SPECS.md
     wording instead of the locked CONTEXT.md decision.
   - Recommendation: Follow D-03 exactly (position-only, no fontSize scale) for independent group
     text; this is a locked decision, not open for research judgment.
