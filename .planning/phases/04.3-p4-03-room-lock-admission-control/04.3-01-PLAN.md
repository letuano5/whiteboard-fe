<objective>
Implement P4-03 room capacity control for saved rooms. Owners can configure participant/editor
caps. The backend decides `baseRole` and `effectiveRole`, rejects over-capacity joins, downgrades
editors when editor capacity is full, and exposes roles in presence so the frontend can show why
editing controls are unavailable.
</objective>

<context>
- Canonical scope: `docs/SPECS.md` `[P4-03] Room capacity control`.
- Acceptance registry: `specs/028-room-lock-admission-control/acceptance.md`.
- Prior phase dependency: P4-02 established `RoomAccessPayload`, share modes, owner-only
  management routes, and server-side mutation authorization using `effectiveRole`.
- Room-level lock/unlock was removed from access semantics; `Room.locked` remains document metadata
  outside the Share/access control surface.
- Realtime queues and auto-promotion after an editor leaves are not required for this phase.
</context>

<success_criteria>

- Share modal exposes capacity controls without a room lock/unlock control.
- Invalid capacity settings are rejected before persistence.
- Participant capacity blocks joins with a clear admission error.
- Editor capacity downgrades eligible editors to viewer on join.
- Presence/online-user data includes `baseRole` and `effectiveRole`.
- Rejoin/reload is sufficient for released editor slots; queue/realtime auto-promotion is not
  required.
  </success_criteria>

<risks>
- Capacity must avoid counting the socket currently rejoining the same room, or reconnects could
  reject themselves.
- The existing P4-02 fallback path for legacy/no-sharing delegates must remain permissive for old
  tests and development setups.
- New Prisma fields require generated client compatibility; tests with mocked DB delegates must
  normalize missing fields safely.
</risks>
