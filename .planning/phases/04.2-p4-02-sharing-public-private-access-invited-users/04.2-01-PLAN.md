# Plan 04.2-01: Implement sharing access modes, invitation management, effective roles, UI modal, and AC tests

<objective>
Deliver P4-02 end to end: saved document owners can manage share links and invited users,
while backend join/mutation paths compute and enforce effective room roles.
</objective>

<tasks>

## T1 - Shared contracts and Prisma schema

**Files:** `packages/shared/src/index.ts`, `backend/prisma/schema.prisma`, new Prisma migration

**Action:**

- Add shared `RoomAccessMode`, `EffectiveRoomRole`, `RoomInvitationSummary`, and expanded
  `RoomAccessPayload` fields for `baseRole`, `effectiveRole`, `visibility`, `shareRevokedAt`,
  and pending invitations.
- Add Prisma fields/indexes for room share revocation and `RoomInvitation` rows keyed by room
  and normalized email.

**Acceptance criteria:** AC-1, AC-3, AC-4, AC-7, AC-8, AC-9

**Verify:** `pnpm --filter @vdt/shared typecheck && pnpm --filter whiteboard-be typecheck`

## T2 - Backend access resolver and share-management HTTP API

**Files:** `backend/src/rooms/room-roles.ts`, `backend/src/rooms/room-sharing.ts`,
`backend/src/app.ts`, `backend/src/auth/runtime-auth.ts`, backend tests

**Action:**

- Resolve room access without auto-creating memberships for unrelated users.
- Claim matching pending invitations for authenticated email users.
- Compute `baseRole` from owner/member/invite/link/public/private and `effectiveRole` after
  lock/capacity downgrade.
- Add owner-only routes to get access summary, set/revoke link mode, invite email, update member
  role, remove member, and revoke pending invites.
- Ensure non-owner management calls return typed forbidden errors.

**Acceptance criteria:** AC-1, AC-2, AC-3, AC-4, AC-6, AC-7, AC-8, AC-9

**Verify:** `pnpm --filter whiteboard-be test:run -- src/rooms/room-sharing.test.ts src/rooms/room-roles.test.ts`

## T3 - Socket enforcement

**Files:** `backend/src/realtime/handlers/join-room.ts`,
`backend/src/realtime/handlers/element-update.ts`,
`backend/src/realtime/handlers/room-role-update.ts`,
`backend/src/realtime/handlers/element-update.test.ts`

**Action:**

- Reject private unauthorized joins before joining the Socket.IO room.
- Emit expanded room access payload after successful join.
- Use `effectiveRole` for all element mutation checks.
- Preserve socket role-update owner checks and add route-level coverage for rejected non-owner
  management attempts.

**Acceptance criteria:** AC-2, AC-5, AC-6, AC-7, AC-8, AC-9

**Verify:** `pnpm --filter whiteboard-be test:run -- src/realtime/handlers/element-update.test.ts src/realtime/handlers/room-role-update.test.ts`

## T4 - Frontend access API and manage-access modal

**Files:** `frontend/src/rooms/room-access-api.ts`, `frontend/src/rooms/ManageAccessModal.tsx`,
`frontend/src/components/ShareLinkButton.tsx`, `frontend/src/rooms/room-access.store.ts`,
frontend tests

**Action:**

- Add typed API functions for access summary, link mode update/revoke, invite, role change,
  member removal, and pending invite revoke.
- Replace copy-only share button with owner-only link controls plus `Manage access` modal.
- Render modal over a dark backdrop and include members, pending invitations, role selects, and
  owner self-protection.
- Hide/disable management actions for non-owner access states.

**Acceptance criteria:** AC-1, AC-2, AC-9

**Verify:** `pnpm --filter whiteboard-fe test -- src/rooms/__tests__/ManageAccessModal.test.tsx src/rooms/__tests__/room-access-api.test.ts`

**Files:** `frontend/src/canvas/Whiteboard.tsx`, `frontend/src/components/toolbar/Toolbar.tsx`,
`frontend/src/canvas/__tests__/Whiteboard.permissions.test.tsx`

**Acceptance criteria:** AC-5

**Verify:** `pnpm --filter whiteboard-fe test -- src/canvas/__tests__/Whiteboard.permissions.test.tsx`

</tasks>

<verification>

- Run AC coverage against `specs/027-sharing-access-invites/acceptance.md`.
- Run targeted backend sharing/access and realtime permission tests.
- Run targeted frontend manage-access and viewer-permission tests.
- Run package typechecks for touched packages.

<risks>

- `resolveRoomAccess` currently auto-creates viewer memberships for any authenticated user; this
  must be changed before private-room denial can be correct.
- Link edit capacity is underspecified before P4-03; context locks capacity as unlimited for this
  phase, with lock still downgrading link editors.
- Existing role-update socket tests may depend on the old single `role` field; preserve backward
  compatibility by keeping `role` as an alias for `effectiveRole`.

</risks>

<rollback>

- The migration is additive except for resolver behavior. If sharing rollout needs rollback, keep
  schema fields and revert route registration/UI entry points first, then restore old resolver
  behavior only if private-room AC coverage is intentionally deferred.

</rollback>

<success_criteria>

- Every `AC-n` has at least one `@covers AC-n` test.
- Unauthorized private joins do not enter the Socket.IO room.
- Non-owner access-management calls are rejected by the backend even if UI is bypassed.
- Viewers cannot mutate via socket and do not see edit controls.
- Revoked share links no longer grant link-derived access.

</success_criteria>
