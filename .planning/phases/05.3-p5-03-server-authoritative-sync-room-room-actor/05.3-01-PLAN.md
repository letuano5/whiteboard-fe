## Scope

## Implementation Slices

1. Add `SyncRoom` backend hot state and serialized executor.
   - Keep state in memory for this phase.
   - Execute critical section as one queued unit.
   - Verification: `pnpm --filter whiteboard-be test -- sync-room`.

2. Add room actor queue factory.
   - Queue by `roomId` so same-room commands serialize and different room commands run
     independently.
   - Verification: AC-tagged concurrency tests in `sync-room.test.ts`.

3. Add in-memory idempotency.
   - Key by actor/request ID for shared P5 commands.
   - Return the original result on duplicate retry without running commit/apply side effects again.
   - Verification: AC-tagged retry test in `sync-room.test.ts`.

4. Export the new backend sync primitives.
   - Keep P5-01 `executeSyncCommand` compatibility entrypoint untouched unless tests require an
     explicit type export.
   - Verification: `pnpm --filter whiteboard-be typecheck`.

## Acceptance-Test Tasks

- `AC-1`: test two concurrent commands submitted to the same room cannot interleave between plan
  and apply.
- `AC-2`: test a blocked command in room A does not prevent room B from committing.
- `AC-3`: test duplicate actor/request retry returns the first result and increments side-effect
  counters only once.

## Risks And Deviation Handling

- Risk: Existing P5-01 legacy commands do not carry stable request IDs. Mitigation: scope
  idempotency to shared P5 command envelopes for P5-03 and keep legacy behavior unchanged.
- Risk: Over-implementing P5-04 conflict behavior. Mitigation: provide a small pluggable planner
  callback for tests and later phases instead of encoding slot conflict policy now.
- If implementation needs a persisted idempotency contract, stop and move that concern to P5-06
  instead of expanding this phase.
